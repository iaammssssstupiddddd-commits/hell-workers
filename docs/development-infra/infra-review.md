## 1. 採用確定コンポーネント

### 1.1 Vertex AI（Gemini + Imagen 4）

- **判断**：採用確定
- **用途**：アセット要件定義のプロンプト生成・メタデータ（ZスロットJSON等）自動生成・2Dコンセプト画像生成
- **追加対応事項**
  - 1アセットあたりのトークン数・画像枚数の前提値を定義する
  - コスト計算式を明示する（例：`入力token数 × 単価 + 画像枚数 × $0.04`）
  - $500 GCPクレジットの費目別上限を先に設計する

### 1.2 GCP g2-standard-4 Spot インスタンス（L4 24GB GPU）

- **判断**：採用確定（上限管理の実装が前提）
- **用途**：TRELLIS / Hunyuan3D のセルフホスト実行基盤
- **追加対応事項**
  - Cloud Billing Budget API によるコスト上限アラートを実装する
  - Spot インスタンスの中断率（経験的に10〜30%/日）を折り込んだ再試行係数をコスト計算に組み込む
  - 1アセット完了ごとにGCSのステートファイルを更新し、中断後に未処理アセットから再開できる冪等設計を実装する

### 1.3 Headless Blender（自動最適化）

- **判断**：採用確定（自動品質ゲートの追加が必須）
- **用途**：ポリゴン削減・原点最適化・PBR→Unlit変換・glTF/GLB出力
- **追加対応事項**
  - 以下を自動チェックするCIゲートを実装する
    - ポリゴン数が上限以内か
    - UV展開が存在するか
    - マテリアル数が規定値以内か
    - スケールが Bevy の単位系と整合しているか
    - 原点がモデル底面（接地部）に設定されているか
  - チェックに落ちたアセットは自動Rejectし、ログに記録する
  - Unlit変換（`KHR_materials_unlit`）の処理をスクリプトに組み込む

## 2. 条件付き採用（PoC 先行）

### 2.1 Tripo AI Pro（キャラクターアセット生成）

- **判断**：条件付き採用（利用規約確認が前提）
- **用途**：クワッドトポロジ・Tポーズ出力・自動リギングを活用したキャラクター生成
- **確認必須事項**
  - [platform.tripo3d.ai/docs/billing](https://platform.tripo3d.ai/docs/billing) で一次情報を確認し、以下を文書化する
    - 商用利用の可否と条件（無料プランは CC BY 4.0 のため有料プランが前提）
    - 出力（Outputs）の第三者提供・競合サービス利用の禁止条項の有無
    - 利用規約変更時のリスク
  - クワッドトポロジ・Tポーズ出力の仕様が一次情報で裏取りできているか確認する（現状は比較サイト経由の情報）

### 2.2 TRELLIS vs Hunyuan3D-2.1（環境アセット生成）

- **判断**：Phase 2 PoC は TRELLIS 優先、Hunyuan3D は法務確認後にオプション追加

| 比較軸 | TRELLIS | Hunyuan3D-2.1 |
|:---|:---|:---|
| 必要 VRAM | 16GB以上（L4で成立） | フル機能で29GB（L4では不足の可能性あり） |
| ライセンス | MIT相当（要一次確認） | 独自ライセンス・EU/UK/韓国で地域制限あり |
| テクスチャ品質 | 良好 | 高品質 |
| Phase 2 推奨 | **優先採用** | 法務確認完了後に検討 |

- **Hunyuan3D-2.1 の確認必須事項**
  - 配布想定地域（EU・UK・韓国等）と地域制限条項が抵触しないか法務レビューを実施する
  - 抵触する場合は TRELLIS または他の MIT ライセンスモデルを代替として固定する
  - L4（24GB）での低VRAMモード（`--low_vram_mode`等）の実際の生成品質をPoC で実測する

## 4. 最優先未解決事項：アートスタイル整合

**この仕様が固まるまで PoC の合格基準を定義できず、量産移行の判断ができない。**

### 4.1 問題の所在

`world_lore.md` が要求するアートスタイルと、AI生成3Dアセットの典型的な出力には大きな乖離がある。

| 要件（world_lore.md） | AI生成の典型出力 | 対応難度 |
|:---|:---|:---:|
| ルーズなインクアウトライン | スムーズなメッシュエッジ | 高 |
| 意図的に歪んだシルエット | 幾何学的に正確な形状 | 高 |
| 筆跡・塗りムラのある質感 | PBRテクスチャ | 中 |
| 黒く歪んだ木（肋骨質感） | 自然な木のテクスチャ | 高 |

### 4.2 技術的解決方針（決定が必要）

以下の2点を PoC 前に決定し、受入基準として文書化する。

**① PBR → Unlit 変換方針**
- Headless Blender の Shader ノードで PBR マテリアルを Shadeless に変換する
- `KHR_materials_unlit` 拡張として glTF エクスポートする
- Bevy の `StandardMaterial` で `unlit: true` を設定して受け取る

**② アウトライン表現の実装方針**（どちらかを選択）
- **案A（Blenderベイク）**：Blender側でアウトラインをテクスチャにベイクして出力する
- **案B（Bevyシェーダー）**：Bevy の WGSL カスタムシェーダーでアウトライン描画を追加する

### 4.3 PoC の合格基準（案）

PoC で検証すべき品質基準を事前に定義しておく。

- `world_lore.md` のアートスタイルガイドに照らして目視確認できること
- Unlit 変換後に「手描き感」が破綻していないこと
- 正射影カメラ（`OrthographicProjection`）でのトップダウン表示で違和感がないこと
- 1アセットあたりの手修正時間が許容範囲内であること（目安：X時間以内）

## 5. Bevy 統合の詳細設計（提案書の過小評価部分）

提案書の「GLBを assets/ に置けば完了」という記述は既存アーキテクチャの複雑さを過小評価している。以下の設計が別途必要。

### 5.1 既存アーキテクチャとの整合

- `VisualLayerKind`（`hw_visual::layer`）の子エンティティ構造に glTF モデルを統合する
- `RenderLayers::layer(1)` を `VisualLayerKind` 子エンティティに付与して Camera3d 側で描画する
- `WallVisualHandles` 等の専用 Resource への startup 注入パターンに従う

### 5.2 壁接続方式の設計決定（未決）

`wall_connection.rs` の16+バリアント廃止に向けて、以下のどちらを採用するか決定が必要。

- **案A（パーツ組み合わせ）**：直線・コーナー・T字・十字等のパーツモデルをセットで用意し、Bevy 側で動的に組み合わせる。この場合 AI に生成させるのは「一体型オブジェクト」ではなく「接続パーツのセット」になる。
- **案B（プロシージャルメッシュ）**：Bevy 側でメッシュを動的生成し、テクスチャのみ AI 生成で賄う。

**この設計が決まるまで、AI に何を生成させるかが定まらない。**

## 6. コストモデルの補強（提案書の欠落部分）

現在の提案書はコストの「方向性」しか示していない。以下の式と上限を定義する必要がある。

| 費目 | 計算式 | 上限 |
|:---|:---|:---|
| GPU計算（自前ホスト） | `GPU単価($/h) × 稼働h × 再試行係数` | $XXX |
| LLM（メタ生成） | `入力token数 × 単価 + 出力token数 × 単価` | $XXX |
| 画像生成（コンセプト） | `画像枚数 × $0.04` | $XXX |
| SaaS（Tripo） | `月額 + 追加クレジット` | $20/月 |
| Blender自動処理 | `CPU単価 × 処理時間（CI環境）` | $XXX |

- Spot 単価は需給変動するため、レンジで計算する（楽観・標準・悲観）
- 月次でコスト実績を確認し、$500 クレジット枠の消化ペースを管理する仕組みを用意する

## 8. PoC の合格条件（最小定義）

以下を全て満たした場合にのみ量産フェーズへ移行する。

- [ ] アートスタイル：`world_lore.md` の指針に照らして目視合格
- [ ] 技術：glTF 取込・RtT 表示・Camera3d 正射影での表示が成立
- [ ] 品質：Blender 自動最適化後のアセットが自動品質ゲートを通過
- [ ] 速度：1アセットあたりの生成〜Bevy 取込までの所要時間が許容範囲内
- [ ] 法務：Tripo AI・TRELLIS の利用条件が配布想定地域で問題ない
- [ ] コスト：PoC 実績に基づくコストがプロジェクト予算と整合する
