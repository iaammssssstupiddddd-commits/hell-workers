## 1. 目的

- 解決したい課題: `docs/plans/README.md` と実ファイルのズレが起きやすく、保守時に手動更新漏れが出る。
- 到達したい状態: plans/proposals の index を半自動 or 自動で生成できる運用を整え、ドキュメント整合性を維持する。
- 成功指標:
  - index 生成スクリプト（または更新コマンド）が追加される。
  - README 一覧が実ファイルと一致する。
  - `cargo check` 成功（Rustコード変更がない場合も確認実施）。

## 2. スコープ

### 対象（In Scope）

- `docs/plans/README.md` の更新ルール整備。
- 必要に応じて `scripts/` に index 更新スクリプトを追加。
- `docs/proposals/README.md` への横展開（必要時）。

### 非対象（Out of Scope）

- 全 docs の自動生成システム化。
- CI ワークフローの大規模改修（任意追補）。

## 3. 現状とギャップ

- 現状:
  - README の一覧は手動更新で運用。
  - アーカイブ配置と現行配置の反映が漏れやすい。
- 問題:
  - 初見時に「どれが現行計画か」が判別しづらい。
- 本計画で埋めるギャップ:
  - index 生成の明確な手順または自動化スクリプトを提供し、更新漏れを減らす。

## 4. 実装方針（高レベル）

- 方針:
  - まずは軽量スクリプト（例: shell/python）で README 表を生成。
  - 既存 README 体裁を維持し、差分は最小限にする。
- 設計上の前提:
  - `docs/plans/` の現行/アーカイブ区分を維持。
  - 手書きの補足文は残せる設計にする。
- Bevy 0.18 APIでの注意点:
  - Rust 実装には影響しないが、作業完了時は `cargo check` を実施する。

## 6. リスクと対策

| リスク | 影響 | 対策 |
| --- | --- | --- |
| 自動生成が既存手書き内容を壊す | README 可読性低下 | 生成対象をセクション限定し保護コメントを導入 |
| ステータス判定誤り | 誤った一覧表示 | ファイル名・front matter 依存を避け明示ルール化 |

## 7. 検証計画

- 必須:
  - `cargo check`
- 手動確認シナリオ:
  - スクリプト実行前後で README 一覧が実ファイルと一致する。
  - アーカイブに移動したファイルが現行一覧から外れる。
- パフォーマンス確認（必要時）:
  - 不要。

## 8. ロールバック方針

- どの単位で戻せるか:
  - スクリプト導入コミットと README 反映コミットを分離して戻せる。
- 戻す時の手順:
  - 問題があればスクリプト導入を revert。
  - README を手動更新運用に戻す。
