提案書：フルRtTおよび多層階アーキテクチャへの移行戦略
日付: 2026-03-14 (Updated)
ステータス: Draft
対象領域: 描画アーキテクチャ、グリッドシステム、入力判定

1. 背景と目的
現在のプロジェクトでは、2Dコードベースの安定性を維持しつつ、建築物（壁など）の表現力を向上させるために「ハイブリッドRtT（Render-to-Texture）」の導入を進めている。
しかし、将来的に「キャラクターの3D化」および「2階建て以上の多層階建築」を実装する構想が確定したため、ハイブリッドアプローチが抱えるZソート（前後関係）の破綻リスクが顕在化する。

本提案では、最終的なゴールを**「地形を含めたインゲーム要素の完全3D化＋2D UIオーバーレイ（フルRtT）」**と再定義し、既存の堅牢なECSロジック層を破壊せずに安全に移行するための段階的なロードマップを定義する。

2. アーキテクチャの基本方針
ゲームの「頭脳（ロジック）」と「見た目（ビュー）」を完全に分離し、描画層のみを3Dにすげ替えるアプローチを取る。

2.1 描画層：完全3D化（Camera3d）とフルRtT
フル3D空間配置: 地形（テレイン）、建築物（壁）、キャラクターを含むすべてのインゲーム要素を3D空間にMeshとして配置する。

平行投影（Orthographic Projection）の採用: Camera3dの投影方式を「平行投影」に設定することで、現在の2Dドット絵と同じ遠近感のないトップダウン視点、および矢視（立面図/クォータービュー）を完全に再現する。

フルRtT合成: 3D空間の描画結果を画面解像度サイズのテクスチャに焼き付け、従来通り2DのUIレイヤーと合成する。

2.2 ロジック層：2Dグリッドの維持と拡張
現在の2Dグリッドシステム（AI、パスファインディング、物流）は、画面がどう描画されているかを関知しないため、基本構造は無改修で維持する。

ただし、将来の多層階対応を見据え、プリミティブな座標型をカプセル化（例：TileIndex(x, y, z) への将来的な拡張を前提とした構造体の導入）し、階層間の移動（階段）を許容するネットワークグラフ設計への準備を進める。

3. 解決される課題（メリット）
Zソート問題の完全解決: すべての要素が3D空間に存在するため、ハードウェアのZバッファにより、壁・キャラクター・地形の前後関係や隠蔽（オクルージョン）がピクセル単位で自動的に解決される。

多層階表現の容易化: 「1階と2階を画面の完全な切り替えで表現する」仕様において、指定した階層（Floor ID）以外の3Dメッシュを非表示にするだけで描画の切り替えが完結する。

ロジックの純粋性保護: 描画のZ値（0.0〜11.0）管理から解放され、ロジックコンポーネントは純粋な論理座標のみを扱えるようになる。

4. 段階的移行ロードマップ
ビッグバン・リファクタリングによる開発停滞を防ぐため、以下のフェーズに分けて移行を実施する。

Phase 1: インフラ準備（既存提案通り）
Camera3d のセットアップ（平行投影設定）。

Render-to-Texture (RtT) パイプラインの構築とUIレイヤーの合成テスト。

Phase 2: ハイブリッドRtT（壁とキャラクターの先行3D化）
地形は2Dレイヤーに残し、「建築物（壁）」と「キャラクター」をセットで3D空間（RtTレイヤー1）に配置する。

これにより早期にZソート問題を解消し、3D Transform変換のノウハウを蓄積する。

Phase 3: フルRtT到達（地形の3D化と入力判定の刷新）
地形の描画ロジックを2D Spriteから3D平面Meshに置き換える。

[最重要改修] マウスクリック判定（Hit Testing）を、2Dの逆行列変換から「Camera3dからのRaycasting（光線交差判定）」へ全面的に書き換える。

2DのインゲームSprite資産を完全に廃止。

Phase 4: 多層階ロジックの導入（将来構想）
ロジック層の座標系に「Z軸（Floor ID）」を導入し、型エラーを修正しながらシステム全体を3D配列対応に引き上げる。

パスファインディングを多層階ネットワーク（階段ポータル）に対応させる。

5. 技術的課題と対策（リスク管理）
6. 結論
最終目標を「フルRtT」に設定することで、ハイブリッド方式の限界（Zソート問題）を根本から解決し、将来の多層階アーキテクチャへの最も確実な布石となる。
ロジック層と描画層が綺麗に分離された現在のECS設計であれば、Phase 1〜3の段階的アプローチにより、安全かつ着実な完全3D描画への移行が可能である。
