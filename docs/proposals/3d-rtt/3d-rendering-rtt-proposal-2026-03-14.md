## 1. 背景と問題

- 現状:
  - 建築物は2Dスプライトの静的組み合わせで描画（壁は16+バリアントのテクスチャ切替）
  - キャラクターは状態別の2Dスプライト切替（ソウル: 7状態、ファミリア: 4フレーム）
  - テレインボーダーはEdge/Corner/InnerCornerスプライト × 3地形 = 9枚のオーバーレイで描画
  - 全描画が`Camera2d` + Z-ordering（21定数、0.0〜11.0）で構成されている

- 問題:
  - **壁の組み合わせ管理が煩雑**: 接続パターンごとに2Dスプライトを用意（角、T字、十字、端等16+バリアント）。新しい壁タイプの追加ごとにバリアント数が乗算的に増加
  - **建築物の見た目が静的**: 作業アニメーション（建設、修理等）が2Dスプライト切替に限定され、動きの表現力に限界がある
  - **矢視（立面図）が実現不可能**: 現在のトップダウン2Dでは建築物を側面から見ることができない
  - **テレイン境界が90度ベース**: ボーダーオーバーレイがタイル単位のため、地形遷移が直角的で不自然

- なぜ今やるか:
  - 建築要素（壁、床、家具、配管、配線等）の追加が進んでおり、2Dバリアント管理の複雑さが加速的に増加している
  - ゲームプレイ上、建築物を側面（矢視）から確認する需要がある
  - Bevy 0.18が`RenderTarget::Image`、`RenderLayers`、3D正射影カメラを標準サポートしており、技術的に実現可能なタイミングである

## 2. 目的（Goals）

- 建築物・キャラクターの描画を3Dモデルベースに移行し、ビジュアルの動的生成を実現する
- 建築物の矢視（4方向固定角度の立面図切り替え）を実装する
- 壁バリアント管理の廃止（3Dモデルの動的配置で代替）
- テレインの90度境界制約を解消し、有機的な地形遷移を実現する
- 既存の2Dコードベースを破壊せず、段階的に移行可能なアーキテクチャとする

## 3. 非目的（Non-Goals）

- 全面的な3D移行（Camera2dの廃止）は行わない
- パースペクティブ（透視投影）カメラの導入は行わない
- 3Dモデルの制作ワークフロー策定（モデリングツール選定、アートパイプライン等）は本提案の範囲外
- PBRライティング・シャドウの全面適用（将来の拡張として位置づけ）
- WFCの地形生成ロジック変更（別提案 `wfc-terrain-generation-plan-2026-03-12` で対応）

## 4. 提案内容（概要）

- 一言要約: **ハイブリッドRtT** — Camera2d（2D）とCamera3d（3D）をRenderLayersで分離し、3DシーンをRender-to-Textureで2Dに合成する
- 主要な変更点:
  1. Bevyの`3d`フィーチャーを有効化し、Camera3d + RenderTargetのインフラを構築
  2. 建築物をRenderLayers::layer(1)の3Dモデルとして描画し、RtTで2Dビューに合成
  3. 矢視モードではCamera3dの角度を4プリセットに切り替え
  4. テレインは2Dのまま維持し、シェーダーブレンディングで90度境界を解消
- 期待される効果:
  - 壁16+バリアントの2Dテクスチャ管理が不要に
  - 建築物の組み合わせ・作業アニメーションがリッチに
  - 同じ3Dモデルでトップダウンと矢視の両方に対応
  - テレインの地形遷移が自然で有機的に

## 5. 詳細設計

### 5.1 アーキテクチャ: ハイブリッドRtTレイヤー構成

```
レイヤー0 (Camera2d):
  テレイン、2Dエフェクト（プログレスバー、吹き出し、ダメージ数値）、
  パーティクル（夢泡等）、RtT合成テクスチャ、UI

レイヤー1 (Camera3d → RenderTarget::Image):
  建築物3Dモデル、キャラクター3Dモデル（オプション）
```

**描画順序:**
1. Camera3d（order: -1）がレイヤー1の3Dモデルを正射影で描画 → オフスクリーンテクスチャ
2. Camera2d（order: 0）がレイヤー0を描画（テレイン → RtTテクスチャスプライト → 2Dエフェクト）
3. UIはBevy標準UIレンダリングで最上位に描画

### 5.2 通常モード（トップダウン）

- Camera3dが3Dモデルを**真上から**正射影で描画
- Camera2dのPanCamera操作に**Camera3dを毎フレーム同期**:
  - パン: `Camera2d.Transform.translation.xy` → `Camera3d.Transform.translation` へ軸マッピング
  - ズーム: `Camera2d.OrthographicProjection.scale` → `Camera3d.OrthographicProjection.scale` へ変換
    （注: `area` は Bevy が自動管理する `Rect` フィールドのため、ズーム制御には `scale` を使用すること）
- RtTテクスチャはフルスクリーンSpriteとしてCamera2dの適切なZ位置に配置
- クリアカラーを透明（`ClearColorConfig::Custom(Color::NONE)`）に設定し、建築物がない部分はテレインが透過して見える

### 5.3 矢視モード（立面図）

- ユーザーがキー/UIで方向を選択すると、Camera3dのTransformを4プリセットの1つに切り替え:
  ```
  正面（南→北）: Transform::from_xyz(0, -d, h).looking_at(target, Vec3::Y)
  背面（北→南）: Transform::from_xyz(0,  d, h).looking_at(target, Vec3::Y)
  左側（西→東）: Transform::from_xyz(-d, 0, h).looking_at(target, Vec3::Y)
  右側（東→西）: Transform::from_xyz( d, 0, h).looking_at(target, Vec3::Y)
  ```
- テレイン（レイヤー0の2Dスプライト）は非表示に切替
- UIオーバーレイはそのまま表示

### 5.4 テレインの90度境界解消

テレインは3D化せず、2D Material2dシェーダーブレンディングで改善する:

- 各タイルのシェーダーに自タイル + 周囲8タイルの地形タイプを渡す
- シェーダーが距離ベースでテクスチャ間をアルファブレンド
- ノイズ関数（Perlin等）を加えて遷移境界を有機的な曲線に
- 論理上のTerrainTypeグリッドは変更なし（80×80のまま）
- 現在のボーダーオーバーレイスプライト（9枚 × 回転）は廃止可能

**代替案: RtTベイク**（セクション6で比較）

## 6. 代替案と比較

### 建築物の描画方式

| 案 | 採否 | 理由 |
| --- | --- | --- |
| ハイブリッドRtT（Camera2d + Camera3d RenderLayers分離） | **採用** | 既存コード変更ゼロ、段階的移行可能、矢視対応 |
| Orthographic 3D全面移行（Camera2dを廃止） | 不採用 | 80+ファイル・10,000+行の書き換え。`Sprite`は`Camera3d`で描画されず全面移行が必要。リスクが非常に高い |
| 2Dスプライトのまま矢視用スプライトを別途用意 | 不採用 | 矢視は実現可能だがアセット二重管理。動的組み合わせ・アニメーションの改善なし |

### テレインの描画方式

| 案 | 採否 | 理由 |
| --- | --- | --- |
| シェーダーブレンド（2D Material2d） | **採用（推奨）** | 低コスト、90度境界解消、ロジック不変、デバッグ容易 |
| RtTベイク（生成時のみ3D→2D画像保存） | 検討中 | ゲーム実行時負荷ゼロ、最も自然な地形。ただしTerrainType逆算のデバッグコストあり |
| 高解像度データグリッド（320×320サブセル） | 不採用 | WFC探索空間16倍増、パスファインディング変更が必要 |
| リアルタイムRtT（毎フレーム3D描画） | 不採用 | テレインにZ軸不要。GPU負荷が不釣り合い |

### テレインRtTベイクについての補足

RtTベイクは「生成時のみ3Dを使い、ゲームでは2D画像のみ使用」する方式:
1. マップ生成時に3Dノイズ地形をCamera3dで真上から正射影レンダリング → テクスチャにベイク
2. ベイクされた2D画像をゲームの地形表示に使用（3Dは使わない）
3. テクスチャの各タイル領域をサンプリングし、最大成分からTerrainTypeを決定

**メリット**: 3D高さマップからの自然な地形分布（高地=草、低地=砂、最低地=川）、有機的な遷移
**デメリット**: テクスチャ→TerrainType逆算ロジック、GPU→CPU読み戻し（生成時1回のみ）、デバッグ困難

**推奨**: まずシェーダーブレンドで90度境界を解消し、地形の豊かさが不足する場合にRtTベイクを追加検討

## 7. 影響範囲

- ゲーム挙動: 変化なし（描画方式の変更のみ、ゲームロジック不変）
- パフォーマンス:
  - 追加のレンダーパス1回分（フルスクリーンRtT）。ローポリ3Dモデルなら軽量
  - テレインの2Dスプライトバッチングは維持
  - 壁バリアントテクスチャメモリは削減の可能性
  - `3d`フィーチャー追加でコンパイル時間30〜60秒増
- UI/UX:
  - 矢視モードの新規UIが必要（方向切替ボタン/キーバインド）
  - 既存UIは一切変更なし
- セーブ互換: 影響なし（描画のみの変更）
- 既存ドキュメント更新: `docs/architecture.md` にレンダリングレイヤー構成を追記

## 8. リスクと対策

| リスク | 影響 | 対策 |
| --- | --- | --- |
| Camera2d↔Camera3d同期のズレ | 中: パン/ズーム時に3Dオブジェクトが2Dと位置ズレ | 毎フレーム同期システム。正射影なので変換は線形で単純 |
| 3Dモデル制作コスト | 中: ローポリでもモデリング作業が必要 | まず壁セグメント1つでプロトタイプ検証。Tim Burton風テクスチャはUnlitマテリアルで対応 |
| RtT透過合成の品質 | 低: アルファブレンド時のアーティファクト | Bevy標準のアルファブレンディング。問題発生時はpremultiplied alpha対応 |
| Bevy 0.18のRenderTarget API安定性 | 低: API変更の可能性 | 標準exampleで動作確認済みのパターンを使用 |
| コンパイル時間増加 | 低: `3d`フィーチャーで30〜60秒増 | 許容範囲。`3d_api`（PBRなし）で最小化も可能 |
| 既存コードへの影響 | **なし** | 完全追加的な変更。いつでも`3d`フィーチャーを外して元に戻せる |

## 10. ロールアウト/ロールバック

- 導入手順:
  1. Phase 1: RtTインフラ + カメラ同期
  2. Phase 2: テスト用簡易3Dモデルで描画確認
  3. Phase 3: 壁システムの3D化（動的配置ロジック）
  4. Phase 4: 矢視（立面図）の4方向切替 + テレインシェーダーブレンド
  5. Phase 5: 他の建築物カテゴリ・キャラクターを段階的に3D化
- 段階導入の有無: **あり**。各Phaseが独立しており、任意のPhaseで停止可能
- 問題発生時の戻し方:
  - Phase 1は完全追加的。`Cargo.toml`から`"3d"`を削除し、追加ファイルを削除するだけ
  - Phase 2以降も、各建築物カテゴリは個別に2Dスプライトに戻せる（レイヤー0に戻すだけ）

### ブロッカー/注意点

- **Bevy 0.18 API厳守**: AIの学習データにある古いAPIを使わないこと（CLAUDE.mdルール5）
- **Bevy 0.18 `RenderTarget`は別コンポーネント**: 旧来の `Camera { target: RenderTarget::Image(handle) }` は廃止済み。
  正しい書き方: `commands.spawn((Camera3d, RenderTarget::Image(render_target_image.into()), ...))`
  `render_target_image.into()` で `Handle<Image>` → `ImageRenderTarget` へ変換が必要
- **Bevy 0.18 `ClearColorConfig`**: カメラのクリアカラーは `Camera` コンポーネントの `clear_color` フィールドで設定:
  `Camera { clear_color: ClearColorConfig::Custom(Color::NONE), ..default() }`
- **3Dモデルが未制作**: Phase 2以降の検証にはテスト用3Dモデル（Bevyの組み込みシェイプ等）を使用
- **PanCameraは2D専用**: Camera3dの同期は独自実装が必要
- **`Sprite`は`Camera3d`で描画されない**: これがハイブリッドRtTを選択した根本的な理由。全面3D移行は80+ファイルの書き換えが必要
