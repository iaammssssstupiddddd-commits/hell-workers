## 1. 背景と問題

- **現状**: `crates/bevy_app/src/world/map/boundary.rs` にて、Catmull-Rom スプラインと法線ノイズで境界ポリラインを計算し、それに幅を持たせた **リボンメッシュ（3Dポリゴン帯）** を生成して地形タイルの上に配置している。
- **問題**: 
  - 48wu という太いリボンメッシュが3D空間上で鋭角に曲がったり、三叉路で密集したりすることで、**極端なジオメトリのねじれ、自己交差、アルファブレンドの二重掛け（干渉・Zファイティング）** が発生し、見た目が著しく崩れるバグの温床となっている。
  - 地形とは別に数千〜数万ポリゴンの半透明メッシュを描画するため、遠景表示（LOD）時のパフォーマンス最適化の障害となっている。
- **なぜ今やるか**: 地形生成・境界算出ロジックは安定しているものの、最終的な3D描画表現のレイヤーで破綻が起きており、アートスタイル（Rough Vector Sketch）の実現とパフォーマンス両面で根本的な解決が急務であるため。

## 2. 目的（Goals）

- 既存の境界ポリライン算出ロジック（Catmull-Rom、ノイズ変位等）を完全に活かす。
- リボンメッシュの生成を廃止し、**「境界マスクテクスチャ」へのピクセル描画（ペースト）** に置き換えることで、幾何学的なねじれや干渉バグを物理的に根絶する。
- マップ全体をフラットな1枚（またはチャンクごと）の地表として描画可能にし、パフォーマンスとアートスタイル（手描きの切り貼り感）の親和性を高める。

## 3. 非目的（Non-Goals）

- WFC（Wave Function Collapse）による地形タイル生成アルゴリズム自体の変更。
- 既存のポリライン算出ロジック（ノイズ、三叉路処理など）の根本的な再設計。
- 境界線の動的（毎フレーム）な形状変化（初期生成時の一括ベイクを想定）。

## 4. 提案内容（概要）

- **一言要約**: リボンメッシュ（ポリゴン）を廃止し、起動時にポリラインから「境界マスクテクスチャ（SDFまたは2Dラスタライズ）」をCPUで生成・ベイクして、地形シェーダーでテクスチャとしてペーストする。
- **主要な変更点**:
  - `boundary.rs` の `spawn_boundary_meshes`（メッシュ生成）を削除。
  - 代わりに、ポリラインデータからマップ全体を覆う高解像度の **境界マスクテクスチャ（あるいは SDF: 符号付き距離場 テクスチャ）** を CPU で生成し、`Image` アセットとして登録する。
  - `TerrainSurfaceMaterial`（地形シェーダー）にこのマスクテクスチャを渡し、シェーダー内でマスクを参照して境界色（ブレンド）をフラットに描画する。
- **期待される効果**:
  - ポリゴン交差による干渉・ねじれ・Zファイティングの完全消滅。
  - リボンメッシュのドローコール・ポリゴン数の削減による描画負荷の大幅低下。
  - 交差点や鋭角がピクセルレベルで綺麗に融合（ペースト）されるため、アートスタイル要件を完全に満たす美しい境界線が得られる。

## 5. 詳細設計

### 5.1 仕様

- **振る舞い**:
  1. `extract_boundary_edges` から `BoundaryPolyline` を生成する処理までは既存のものを流用。
  2. CPU 側でマップ領域全体をカバーするテクスチャバッファ（例: 2048x2048 または 4096x4096、マップサイズに依存）を確保。
  3. 各ポリラインをバッファに対して描画する（**SDF 距離場計算** または **2Dラスタライズライブラリによる太線描画**）。
  4. 生成したバッファを `Image` として GPU へ転送し、`TerrainSurfaceMaterial` のバインディング（新規）にセット。
  5. `terrain_surface_material.wgsl` で、ワールド座標から UV を算出してマスクテクスチャをサンプリング。閾値を超えたピクセルに境界のテクスチャ（草や土など）をペーストする。
- **例外ケース**: マップサイズが巨大な場合、テクスチャサイズが大きくなりすぎる可能性がある。その場合はテクスチャの解像度を下げるか（SDFなら低解像度でも綺麗に補間可能）、複数枚に分割する。
- **既存仕様との整合**: 既存の `terrain_id_map` との併用が可能。マスクテクスチャに境界種別（BoundaryKind）のIDを持たせることで、シェーダー側でどの地形テクスチャを引っ張ってくるかを判断できる。

### 5.2 変更対象（想定）

- `crates/bevy_app/src/world/map/boundary.rs`（メッシュ生成処理の削除、テクスチャ生成処理の追加）
- `crates/hw_visual/src/material/terrain_surface_material.rs`（マスクテクスチャのバインディング追加）
- `assets/shaders/terrain_surface_material.wgsl`（マスクテクスチャのサンプリングとペースト処理）
- `crates/hw_visual/src/material/boundary_surface_material.rs`（完全に廃止可能）

### 5.3 データ/コンポーネント/API 変更

- **追加**:
  - `BoundaryMaskMap` (Resource): 生成された境界マスクテクスチャのハンドルを保持。
- **変更**:
  - `TerrainSurfaceMaterialExt` に `boundary_mask: Option<Handle<Image>>` などを追加。
- **削除**:
  - `BoundarySurfaceMaterial`, `BoundaryMeshMarker`, `spawn_boundary_meshes` システム。

## 6. 代替案と比較

| 案 | 採否 | 理由 |
| --- | --- | --- |
| **案A: 2Dラスタライズ (CPU)** | 採用候補 | `tiny-skia` 等を用いてポリラインを太線としてCPU描画する。最も直感的で「絵をペーストする」感覚に近い。線の結合も綺麗。 |
| **案B: SDF (符号付き距離場) 生成** | **推奨採用** | ポリラインからの距離をテクスチャに書き込む。低解像度（例: 1024x1024）のテクスチャでもシェーダー側の補間で極めて滑らかで美しい境界が描け、交差部が数学的に綺麗に融合する（メタボール効果）。CPU負荷も抑えやすい。 |
| 案C: GPU ステンシル/深度テスト | 不採用 | メッシュのねじれ自体は残るため、極端なジオメトリによる根本的な問題解決には至らない。 |

本提案では、**案B（SDF生成）** または **案A（2Dラスタライズ）** のいずれかを実装方針として採用することを前提とする。

## 7. 影響範囲

- **ゲーム挙動**: 影響なし（純粋なビジュアル変更）。
- **パフォーマンス**: 大幅な向上。数千の半透明メッシュ描画が削減され、LODへの移行も容易になる。起動時のテクスチャ生成にわずかな時間（数十〜数百ms）が追加される。
- **UI/UX**: 見た目が大幅に改善される。干渉やチラつきがなくなる。
- **セーブ互換**: 影響なし。
- **既存ドキュメント更新**: `docs/world_layout.md`, `docs/art-style-criteria.md` のレンダリング方針に関する記述の更新が必要。

## 8. リスクと対策

| リスク | 影響 | 対策 |
| --- | --- | --- |
| マップ起動時間の増加 | 起動時に CPU で画像（SDFまたはラスタライズ）を生成するため、ロード時間が長くなる可能性がある。 | 解像度を適切に落とす（SDFなら低解像度でも高品質）。必要であれば非同期生成（Task）に逃がす。 |
| VRAM 消費の増加 | マップ全体を覆うテクスチャを1枚追加するため、VRAM を消費する。 | SDF形式（1チャンネル）を利用することでデータサイズを最小限に抑える（R8Unorm等）。 |

## 9. 検証計画

- `cargo check --workspace`
- 手動確認シナリオ:
  - ゲームを起動し、境界線（草と土、川と砂など）の描画が干渉・ねじれなくフラットに描画されているか確認する。
  - 三叉路や鋭角に曲がる箇所で破綻（穴あきや不自然な重なり）がないか確認する。
- 計測/ログ確認: 起動時の境界マスク生成にかかる時間をログ出力し、許容範囲内（数百ms以内）であることを確認する。

## 10. ロールアウト/ロールバック

- **導入手順**: 
  1. `BoundarySurfaceMaterial` のスポーンを無効化。
  2. ポリラインからマスクテクスチャを生成するロジックを実装。
  3. `TerrainSurfaceMaterial` シェーダーを更新してマスクを適用。
- **段階導入の有無**: なし（描画パイプラインの入れ替えとなるため一括導入）。
- **問題発生時の戻し方**: git commit を revert し、旧 `spawn_boundary_meshes` の呼び出しを復活させる。
