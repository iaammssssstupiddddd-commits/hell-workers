## 1. 目的

- **解決したい課題:**
  - 現在の地形生成は完全固定レイアウト（`(x+y)%30==0` パターンのDirt、ハードコードされた川）で、バリエーションがない
  - 現在のボーダー描画はモックアップ（優先度ベースのオーバーレイスプライト）であり、3種類以上の地形が1点で接する際の見た目が保証されない
  - 将来的なBlob-47 Autotilingや描画品質改善の前提として、**3点接触が発生しないことを地形生成側で保証したい**

- **到達したい状態:**
  - Wave Function Collapse（WFC）の隣接制約により、意図しない地形組み合わせが生成されない
  - 地形生成にシード値による再現性を持たせつつ、バリエーションを導入できる
  - 描画システムが「地形Aと地形Bの境界のみを考えれば良い」前提で設計できる

- **成功指標:**
  - WFC生成後のマップで、`草/土/砂` が同一ポイントで3種同時接触するセルが存在しない
  - シード値を変えると異なるマップが生成されるが、川・砂・土・草の大局的な構造は維持される
  - `cargo check` が通る

## 3. 現状とギャップ

### 現状

```
mapgen.rs の generate_base_terrain_tiles():
  1. 固定の川帯（y=65-69, 全幅）を配置
  2. 川周辺2タイルに砂を配置
  3. (x+y)%30==0 の対角線パターンに土を配置
  4. 残りすべてを草に設定
```

**問題:**
- バリエーションが存在しない（シード値なし）
- 土の配置が幾何学的パターンのため、現実的な地形に見えない
- 3点接触の制御なし（砂と土が川の角で接触しうる）
- 将来Blob-47を導入する際、全地形ペアのトランジションセットが必要になる（草↔土, 土↔砂, 砂↔川 の3セットで済ませたい）

**本計画で埋めるギャップ:**
- 地形遷移グラフを明示的に定義し、WFCがそれを強制する
- 地形はグラフ上で「隣接」する種類同士しか接しない → 3点接触を構造的に防止

## 4. 実装方針

### 地形遷移グラフ

現在の優先度チェーン（線形）をそのまま隣接ルールに転用する:

```
川 ─── 砂 ─── 土 ─── 草
```

**隣接可能ペア（双方向）:**
| 地形A | 地形B | 備考 |
|------|------|------|
| 川   | 砂   | 川は必ず砂に囲まれる |
| 砂   | 土   | 砂と土は隣接可 |
| 土   | 草   | 土と草は隣接可 |
| 川   | 川   | 同種は常に隣接可 |
| 砂   | 砂   | 同種は常に隣接可 |
| 土   | 土   | 同種は常に隣接可 |
| 草   | 草   | 同種は常に隣接可 |

**禁止ペア（明示）:**
| 地形A | 地形B | 理由 |
|------|------|------|
| 川   | 土   | 川に直接土が接しない（砂バッファ必須） |
| 川   | 草   | 川に直接草が接しない |
| 砂   | 草   | 砂と草は直接接しない（土バッファ必須） |

この制約により、川の周囲には必ず砂が存在し、土を挟まずに砂と草が隣接することもない。

### ⚠️ 4方向ルールだけでは不十分: コーナー制約の必要性

4方向（上下左右）の隣接ルールのみでは、**斜め方向（コーナー）での禁止ペア接触を防げない**。

```
[草][土]
[土][砂]   ← 4方向ルールは全て通過するが、草と砂が斜めで接触している
```

- 草↔土（右）: ✅ 許可
- 土↔砂（下）: ✅ 許可
- **草と砂がコーナーで接触** → `砂↔草` の禁止ペアが角で実質成立

これが「3種が1点で接触する」問題の生成側の原因であり、コーナー制約を追加で定義する必要がある。

### コーナー制約のアプローチ（3択）

#### アプローチ1: 8方向WFC制約（斜め方向にも隣接ルールを適用）

4方向の隣接ルールと同一のチェーンルールを、斜め方向にも適用する。

```
斜め方向の隣接可能ペア（4方向と同一のチェーンルール）:
  川 ↔ 川, 砂
  砂 ↔ 川, 砂, 土
  土 ↔ 砂, 土, 草
  草 ↔ 土, 草
```

**利点:** `bevy_procedural_tilemaps` が8方向ソケットをサポートしていれば、ルール追加のみで対応可能。
**欠点:** 制約が強くなりすぎてWFCが収束しにくくなる可能性がある。川固定＋砂バッファ固定で探索空間を縮小してから適用する。

#### アプローチ2: 2×2ブロック制約（禁止パターンの列挙）

隣接する4セル（2×2）の組み合わせに対して、禁止パターンを直接定義する。

禁止される2×2パターン（斜め対角に禁止ペアが存在するもの）:
```
[川][*]   [*][川]   [川][*]   [*][川]
[*][土]   [土][*]   [*][草]   [草][*]

[砂][*]   [*][砂]
[*][草]   [草][*]
```
（`*` は任意の地形）

**利点:** 制約が明示的で理解しやすい。WFCがサポートしていない場合も後処理として実装可能。
**欠点:** `bevy_procedural_tilemaps` の標準APIに2×2ブロック制約が含まれるか要確認。

#### アプローチ3: 生成後の後処理修復（最も確実）

WFC生成後に全2×2ブロックを走査し、禁止コーナーパターンを検出して修復する。

```rust
// 疑似コード
for y in 0..MAP_HEIGHT-1 {
    for x in 0..MAP_WIDTH-1 {
        let [a, b, c, d] = get_2x2(x, y);  // [TL, TR, BL, BR]
        if is_forbidden_corner(a, b, c, d) {
            // 最もコストの低い修復: d を c か b と同じ種別に変更
            repair_corner(x, y, a, b, c, d);
        }
    }
}

// 禁止コーナー判定: 対角に禁止ペアが存在する
fn is_forbidden_corner(a, b, c, d) -> bool {
    !can_be_diagonal(a, d) || !can_be_diagonal(b, c)
}

fn can_be_diagonal(t1, t2) -> bool {
    // 4方向ルールと同一: チェーン上で隣接している種別のみ許可
    (t1 == t2) || are_chain_adjacent(t1, t2)
}
```

**利点:** WFCライブラリのAPIに依存しない。実装が独立していて検証しやすい。
**欠点:** WFC生成後に別パスが必要。修復により元のWFC解が崩れる可能性があるが、局所修復なので影響は小さい。

### 推奨アプローチ

**アプローチ1（8方向制約）を優先し、収束しない場合はアプローチ3（後処理修復）にフォールバック。**

理由:
- アプローチ1はWFCソルバーが本来持つ伝播能力を活かせる
- 川・砂バッファを固定化することで制約が強くても収束しやすくなる
- それでも収束しない場合、アプローチ3は独立したパスとして確実に機能する

### WFCアプローチ選択

#### 選択肢A: `bevy_procedural_tilemaps` クレート（`ghx_proc_gen` フォーク）

- **Bevy 0.18対応:** ✅（確認済み）
- **利点:** WFCソルバー・隣接制約ビルダーが完備。`SocketCollection` と `RulesBuilder` で上記ルールを宣言的に記述できる
- **欠点:** 外部依存追加。APIが内部実装に干渉する可能性

#### 選択肢B: 独自WFC実装（`hw_world` クレート内）

- **利点:** 依存なし。現在の `WorldMap` データ構造と密結合した最適化が可能
- **欠点:** 実装コスト高。バックトラッキング・エントロピー計算の実装が必要

#### 選択肢C: ハイブリッド（WFCでゾーン生成 → 既存ロジックで詳細配置）

1. WFCで大域的なゾーン（川帯・砂帯・土帯・草帯）のシルエットを生成
2. 既存の `river.rs` / `mapgen.rs` ロジックで詳細を上書き

**推奨: 選択肢C（段階的移行）**
- 初期実装は現在の大域構造（川=水平帯）を維持しつつ、WFC制約で局所的なノイズを生成
- `bevy_procedural_tilemaps` を採用し、WFCソルバーの実装コストを省く

### 設計上の前提

- 川の大域的な位置（水平帯）は `layout.rs` の定数として維持
- WFCは主に「川周辺の砂・土・草の境界形状」の生成に使用
- シード値は `WorldMap` のリソース初期化時に設定

### Bevy 0.18 API での注意点

- `bevy_procedural_tilemaps` はBevy 0.18対応を確認済みだが、API変更に備え`Cargo.toml`でバージョンを固定する
- WFC生成はスタートアップシステムで同期実行（`spawn_map` の前に完了）
- 生成結果は `Vec<TerrainType>` として既存の `WorldMap` 初期化フローに渡す（インターフェースの変更を最小化）

| リスク | 影響 | 対策 |
| --- | --- | --- |
| `bevy_procedural_tilemaps` のAPIがBevy 0.18と非互換 | M2でブロック | `ghx_proc_gen` コアクレートを直接使用するか選択肢Bに切り替え |
